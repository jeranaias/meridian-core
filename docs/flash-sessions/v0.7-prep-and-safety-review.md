# Meridian v0.7 — Pre-Flash Safety Review

Date: 2026-05-04 (continuing the dry-run-day session)
Context: ArduRover currently running on Cube. Tristan stepping away after this attempt. v0.4/v0.5/v0.6 all flashed clean but USB CDC silent. v0.7 is the next attempt; if it fails, no remote power-cycle until USB relay arrives.

## Diagnosis (high confidence)

**Root cause hypothesis:** Cube Orange Plus wires USB VBUS to PA9 as a plain GPIO (per ArduPilot's hwdef.dat), NOT to the OTG VBUS-sense alternate function. `synopsys-usb-otg`'s default H7 init enables VBUS sensing on the OTG core. The core never sees session-valid → never asserts the D+ pull-up → host never enumerates the device.

**Evidence supporting:**
- Symptom matches exactly: device flashes clean, ELF has `.text`/`.vector_table`, boots, host sees nothing.
- ArduPilot's ChibiOS USB driver explicitly forces `GOTGCTL.BVALOEN | BVALOVAL = 1` (B-session-valid override) and clears `GCCFG.VBDEN` (disable VBUS detect). Reference: `libraries/AP_HAL_ChibiOS/USBDriver*.cpp`.
- `synopsys-usb-otg` issue tracker has multiple "fails to enumerate, F4/H7" reports root-caused to VBUS sensing on self-powered designs.
- Cube Plus IS self-powered (POWER1/POWER2 module + AUX BEC), exactly the scenario where forced session-valid is required.

**What's NOT the cause** (already ruled out across v0.4-v0.6):
- Heartbeat plumbing — USB never came up to send heartbeats over.
- USB regulator (`PWR_CR3.USB33DEN`) — added in v0.5, no change.
- USB clock source (HSI48 vs PLL3_Q at 48 MHz) — v0.6 used PLL3_Q, no change.
- Wrong USB peripheral (USB1 vs USB2) — verified USB2 is correct for PA11/PA12 + AF10 on rm0433 H743/H757.
- Wrong PAC features — `stm32h743v` resolves to rm0433 which is correct for H743/H757.
- SYSCLK 480 MHz (known stm32h7xx-hal #503 issue) — we run 400 MHz, not affected.

## v0.7 Changes

### Primary fix: VBUS override
After `UsbBus::new()` (which runs synopsys's `enable()`), patch the OTG2 register block via raw PAC:
- `GOTGCTL` bit 6 (BVALOEN) and bit 7 (BVALOVAL) → 1 to force B-session valid.
- `GCCFG` bit 24 (VBDEN) → 0 to disable VBUS detection.

Located in `crates/meridian-platform-stm32/src/rescue.rs` `usb_cdc::init()`.

### Safety net 1: Hardware watchdog armed BEFORE USB init
Moved `Watchdog::new().start()` from after-TIM6 (was step 7) to immediately before USB init (now in step 4b.5). 2-second IWDG timeout. If USB init hangs, IWDG fires within 2s, Cube resets, bootloader runs, host watcher catches.

### Safety net 2: USB-silence timeout
Added `usb_cdc::has_enumerated()` flag, set true on first `device.poll()` returning true (any host event). Main loop checks every iteration: if `tick > 120000` and not enumerated, the firmware writes the bootloader-magic and resets itself. Guaranteed self-recovery within 2 minutes regardless of how the VBUS fix works.

### Together these guarantee:
| Failure mode | Recovery |
|---|---|
| USB init panics or hangs | IWDG fires in 2s → bootloader → watcher catches |
| USB init succeeds but main loop hangs | IWDG fires → recovery |
| USB init succeeds, main loop runs, host never enumerates | 120s silence timeout → self-reboot to bootloader → watcher catches |
| USB enumerates, all fine | No recovery triggered, normal operation |

**There is no path that locks us out without Tristan.**

## Flash sequence (v0.7)

1. Confirm Cube is currently on ArduRover (2 COM ports = ArduRover, 1 = bootloader, 0 = silent firmware).
2. Update `launch-aggressive-meridian.ps1` to point at `meridian-cubeorangeplus-v0.7-vbus.apj`.
3. Stage v0.7 .apj on USV.
4. Arm watcher with v0.7 target.
5. Send `MAV_CMD_PREFLIGHT_REBOOT_SHUTDOWN param1=3` to ArduRover via `usv-mav-reboot-to-bl.py`.
6. Cube enters bootloader (sticky DFU mode per `param1=3`).
7. Watcher catches at 100ms polling.
8. uploader.py flashes v0.7 (~30s).
9. Cube reboots into Meridian v0.7.
10. Run `usv-find-meridian.py` — looking for VID 0x1209 / PID 0x5741 enumeration.

## What success looks like

- Within ~5s of boot: Windows enumerates a USB CDC device with VID 0x1209 / PID 0x5741.
- Within ~10s: `usv-find-meridian.py` reports "MERIDIAN FOUND".
- `usv-meridian-test.py` shows MAVLink heartbeat coming over USB CDC (autopilot=8 = generic).
- DO_SET_SERVO sweep on ch9 → Tristan sees servo physically move.

## What "graceful failure" looks like

- VBUS fix didn't crack it.
- USB silent at 60s. Watchdog still happy (main loop running).
- At t+120s: firmware self-reboots to bootloader.
- Aggressive watcher was probably ended after the v0.7 flash succeeded. Need to re-arm with ArduRover .apj before this auto-recovery would help.

**ACTION REQUIRED before flashing v0.7:** the watcher will exit after v0.7 flashes (rc=0). If v0.7 fails to enumerate and self-reboots at 120s, no watcher is listening. Need to re-arm watcher (with ArduRover) AFTER v0.7 flash succeeds, BEFORE the 120s timer fires.

Plan: after v0.7 flash success, immediately re-arm watcher with ArduRover .apj and let it sit for 2-3 minutes. If v0.7 enumerates: cancel the watcher, run the smoke test. If v0.7 self-reboots at 120s: watcher catches and restores ArduRover. Either path is recoverable.
